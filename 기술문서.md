# 기술문서 — 이야기가 있는 여행 (Travel with Stories)

작성일: 2026-06-25 · 구현 중심 (아키텍처·데이터·API·실행·유지보수)

## 1. 시스템 개요

단일 서버(모놀리식) 구조다. FastAPI 한 프로세스가 정적 화면과 JSON API를 모두 서빙하고, 데이터는 SQLite에 두며, 질문 응답만 로컬 AI 모델(Ollama)에 위임한다. 별도 캐시·워커는 데모 범위상 두지 않았다.

## 2. 기술 스택

| 영역 | 기술 | 비고 |
|---|---|---|
| 백엔드 | FastAPI (Python) | fastapi / uvicorn |
| DB | SQLite | 서버 시작 시 자동 생성·시드 (설치 불필요) |
| 프론트엔드 | 정적 HTML + 바닐라 JS + CSS | 빌드 도구 없음, 웹/모바일 별도 레이아웃 |
| AI(질문 응답) | 로컬 Ollama (OpenAI 호환 API) | 기본 exaone3.5:2.4b · 요금 없음 |
| 지도 | Google Maps 키 없는 임베드(iframe) | API 키 불필요 |

## 3. 프로젝트 구조

```
seoul_story_bot/
├─ app/
│  ├─ main.py     # FastAPI 라우트
│  ├─ db.py       # SQLite 생성·시드·조회
│  ├─ catalog.py  # 시드 데이터 단일 소스(지역·OTT·연예인·작품·장소)
│  └─ llm.py      # 질문 응답(AI, 근거 주입)
├─ data/
│  ├─ contents.json # 화면 문구(ui) 전용 — 데이터 아님
│  └─ app.db        # 런타임 자동 생성 (gitignore)
├─ static/         # index.html, app.js, style.css
├─ run.bat / run.py · requirements.txt
```

**데이터 단일 소스**: 지역·OTT·연예인·작품·장소는 모두 `app/catalog.py`에 있고, 서버 시작 시 `app/db.py`가 SQLite로 시드한다. `contents.json`은 화면 문구(다국어)만 담는다.

## 4. 데이터 모델 (SQLite)

핵심은 '지역 계층 + 작품을 가운데 둔 다대다 연결'이다.

| 테이블 | 내용 |
|---|---|
| province / city | 도 → 시·구 계층 |
| place | 장소(시에 속함, 좌표 검색어·해설 등) |
| work | 작품(제목·요약·인기순) |
| platform | OTT (넷플릭스·티빙·디즈니+·웨이브) |
| person | 연예인(배우·아이돌) |
| work_platform | 작품 ↔ OTT (M:N) |
| work_person | 작품 ↔ 출연 연예인 (M:N) |
| place_work | 장소 ↔ 작품 + 그 쌍의 해설(story) |

모든 표시 텍스트는 `_ko` / `_en` 두 컬럼으로 저장하고, 조회 시 언어를 골라 준다. '작품의 동선'은 *선택한 시의 장소 중 그 작품이 태그된 것*으로 계산된다.

## 5. 백엔드 API

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | /health | 상태 확인 |
| GET | /api/ui?lang | 화면 문구 |
| GET | /api/provinces?lang | 도 목록(데이터 유무) |
| GET | /api/cities?province=&lang | 시·구 목록(데이터 유무) |
| GET | /api/platforms?city=&lang | OTT 목록 + **그 시의 동선 작품 수** |
| GET | /api/persons?city=&lang | 연예인 목록 + 동선 작품 수 |
| GET | /api/works?city=&platform=\|person=&lang | 작품 목록(인기순, 동선 유무) |
| GET | /api/route?city=&work=&lang | 작품 동선(장소들 + 교차안내) |
| POST | /api/ask | 질문 응답 {city, work, question, lang} |

`/api/platforms`·`/api/persons`는 선택한 시에 **동선이 있는 작품을 가진 것만** 노출하고, 개수도 그 동선 작품 수다.

## 6. 질문 응답 모듈 (app/llm.py)

선택한 작품 동선의 준비된 텍스트(장소 해설)만 근거로 모델에 전달하고, '근거 안에서만 답하고 없으면 정해진 한 문장으로 답하라'고 지시한다(환각 억제). 기본은 로컬 Ollama(요금 없음)이며, localhost 통신은 `httpx trust_env=False`로 프록시를 우회한다(사내·교내망 대응). Ollama가 없으면 빈 화면 대신 안내 문구로 폴백하므로 화면 흐름은 멈추지 않는다. 환경변수 `LLM_BASE_URL`/`LLM_MODEL`/`LLM_API_KEY`로 OpenAI 전환이 가능하다.

## 7. 프론트엔드

프레임워크 없이 `index.html` + `app.js` + `style.css`로 구성한다. 상태 기계로 도→시→OTT/연예인→작품→동선 단계를 렌더하고, 상단 빵부스러기/뒤로 버튼으로 이동한다. 접속 시 화면 폭(1000px 기준)으로 **웹/모바일 별도 레이아웃**을 적용한다(웹: 넓은 그리드·동선 2단, 모바일: 단일 열·세로 스택). 출력 텍스트는 이스케이프 처리하고, 지도는 키 없는 표준 임베드(maps.google.com)를 사용한다.

## 8. 실행 및 배포

```bash
pip install -r requirements.txt
# (선택) ollama pull exaone3.5:2.4b   # 없어도 화면 흐름은 동작
python run.py     # 또는 run.bat / uvicorn app.main:app --host 0.0.0.0 --port 8000
# 브라우저: http://localhost:8000
```

다른 PC는 같은 네트워크에서 `http://<서버 IP>:8000`로 접속한다(방화벽 8000 인바운드 허용 필요할 수 있음). SQLite·표준 라이브러리만 쓰므로 추가 설치는 없다.

## 9. 유지보수

- **데이터 추가·수정**: `app/catalog.py`를 편집한 뒤 **`data/app.db`를 삭제**하고 서버를 재시작하면 새 데이터로 다시 시드된다(시드는 DB가 비어 있을 때만 동작).
- **장소 없는 작품**: 목록엔 보이되 '동선 준비 중'으로 표시 → 장소를 확보하면 동선이 생긴다.
- **정확성**: 촬영지·출연진·OTT 제공처는 공개자료 기반이며 일부는 검증 중(`catalog.py` 주석의 '검토' 표기). 발표 전 1차 확인을 권장한다.

## 10. 알려진 한계

저장소는 단일 SQLite(데모)이고 실행 중 편집은 재시드 방식이다. 위치 자동 인식이 없고(사용자가 다음 지점을 누름), 장소 좌표는 이름 기반 지오코딩을 쓴다. 규모가 커지면 DB를 원본으로 한 관리 화면(CRUD) 도입이 다음 단계다.
